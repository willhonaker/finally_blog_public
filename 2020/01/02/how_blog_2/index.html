How I Made This Blog - Part 2

More adventures in procrastination.

Intro

Want to make a website with Hugo, just like this one? In this post, I’ll outline how I did it, step by step.

Step 0: Conquer your procrastination

This step took me a while. Try not to get distracted, and remember that if you don’t have time for something, you make time for it.

Stop right there!!

Step 1: Choose a static site generator and a theme

As mentioned in the previous post, I wanted a static site generator that could create blazingly fast websites. Google research led me to Hugo.

The next step was to choose a theme. Most static site generators have large libraries of user-created themes, and your choice of generator may come down to which one features themes you like the best. Or you could make your own theme - but that time-consuming task might be better for a future project.

I ended up going with the Clean White theme by Huabing Zhao. Not only did it have all the design elements I wanted - sidebar, tags, site search, social media buttons, etc. - it also came with support for comments (Disqus) and Google Analytics, all built-in and ready to go, which meant less time building and more time publishing. It even looks nice, too.

No need to provide a screenshot of the theme - you're already looking at it.

Step 2: Install Hugo and generate a site

To install Hugo, I followed this guide. But since I’m on a Mac and familiar with Homebrew, all I really had to do was to open a terminal window and run:

brew install hugo

Then:

hugo version

to make sure everything had installed correctly.

Installed successfully.

You’re now ready to generate a site! You can run the following lines to create a starter site and host locally so you can test it out.

hugo new site my_blog
cd my_blog
hugo serve

There are a few different ways to start a blog and incorporate a theme. In my case, I wanted to achieve the following:

  1. Use the aforementioned Clean White theme.
  2. Host on GitHub.
  3. Be able to customize the theme.

The following steps gave me what I wanted:

1. Initializing a new GitHub repo for the site

(The following assumes you have Git installed and linked to a GitHub account.) In a new terminal window:

$ mkdir my_blog && cd my_blog
$ touch README
$ git init
$ git add .
$ git commit -m "New repo for the blog."

The above creates a new directory called my_blog, adds a blank README file inside it, and initializes the directory as a Git repo.

Linking to GitHub is easy:

$ git remote add origin https://github.com/my_github/my_blog
$ git push origin master

Now that the repo’s been created, the next step is to generate the actual blog site.

2. Forking the desired theme

A very simple step: I went to the theme’s github repo and clicked Fork.

That gave me a copy of the theme’s repo to customize as I pleased.

3. Building the site locally and incorporating the forked theme as a submodule

Back to the terminal. To build a site that incorporates the theme, I did the following:

$ mkdir themes && cd themes
$ git submodule add https://github.com/my_github/hugo-theme-cleanwhite.git
$ cp -r hugo-theme-cleanwhite/exampleSite/** ../
$ cd ..
$ hugo serve

The above commands first create a folder for site themes and then adds the forked theme to it as a submodule (ie, as a separate repo that is referenced by this one. See this article for details on how submodules work). This keeps the code bases of the theme and the blog separate, which can make managing change history a little easier.

The cp command then copies over the example site included in the theme into the main repo. cd back into the main repo and run hugo serve to test the site locally.

We are (almost) live!

4. Pushing the locally built site to GitHub for hosting

To put everything back on GitHub:

$ git add .
$ git commit -m "Built the site."
$ git push origin master

The above pushes everything added in step 3 to the main repo. At this point, you could go into the repo’s settings and turn on GitHub Pages to deploy the site live. However, one downside is that the repo must be public, meaning that not just your website but also the backend code powering it is available for anyone to browse - potentially an issue if you’ve got API keys or other credentials coded into your site for features like analytics or site search. I wanted a way to host just the site while keeping the backend private.

The solution? A Netlify app that takes a site built from code in a private repo and publishes it to a separate public repo.

The best part? It’s free. I’ll detail how I did it in the next post.
